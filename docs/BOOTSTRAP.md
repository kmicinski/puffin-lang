# Bootstrapping Puffin: assessment, evaluation, and the feature report

> *(2026-07-10, reader's note: this report is a point-in-time
> document; its history sections stand as written. Since then: the
> "web interpreter" execution route it mentions was retired — the
> browser now runs puffincc itself on the wasm bytecode VM
> (docs/WASM-VM.md), so "all four routes" reads today as reference
> interpreter, x86-64, arm64, and the bytecode VM; the corpus is 300
> checks per route; and puffincc gained the gradual typechecker
> (docs/TYPES.md) and the bytecode backend (docs/BYTECODE.md) beyond
> the self-hosting milestone §5 records.)*

The goal: rewrite the Puffin compiler (today ~5,400 lines of Racket
in `src/*.rkt` + `prelude.puf`, over a ~1,400-line C runtime) in
Puffin itself. This report covers (1) an evaluation of the
Racket-based compiler as it stands, (2) a systematic inventory of
every Racket facility it uses versus what Puffin offers, (3) the
features that inventory demanded, **each now implemented and tested
on all four execution routes** (reference interpreter, x86-64,
arm64, web interpreter), and (4) what still separates us from
self-hosting.

## 1. Evaluation of the Racket compiler

### Correctness infrastructure

- **549 golden checks** per full run (`racket src/test.rkt -m all`):
  61 corpus programs (every class example r0-\*–r5-\* unmodified,
  plus the puffin-\* feature tests) × 3 input files × 3 routes, all
  byte-identical against goldens generated by the reference
  interpreter.
- **Chain mode** (`-m chain`) additionally validates, per program ×
  input, every frontend pass's interpreter output *and* both IR
  predicates — ~2,000 checks; this is what localizes a bug to a pass
  in one command, and it is how three latent class-compiler bugs
  (uniqueify sibling scopes, `revealed?` shadowing, lift-lambdas
  `env` capture) were caught.
- The web interpreter is held to the same goldens
  (`node web/test-corpus.mjs`, 183 checks).

### Performance (Apple M-series, this machine)

> A full nine-benchmark head-to-head suite with plots now lives in
> **`bench/report.html`** (regenerate: `python3 bench/run-benchmarks.py
> && python3 bench/build-report.py`). Headline: geometric-mean wall
> time **1.32× Racket** on arm64; Puffin outright faster on mutable
> hashes, vectors, and strings; Racket ahead on allocation-heavy
> symbolic work (sort 3.4×, meta-interpreter 2.6×, HAMT 2.3×). The
> table below is the earlier spot-check kept for history.

Generated code, `fib(35)` (call-heavy worst case):

| route | time (user) |
|---|---|
| Puffin arm64 native | 0.25 s |
| Puffin x86-64 under Rosetta | ~0.3 s |
| Racket (Chez) | 0.14 s |
| Puffin reference interpreter | ~50–100× native |

Racket wins ~1.8× on call-heavy microbenchmarks (Chez's calling
convention and arithmetic are decades tuned; ours round-trips every
value through the accumulator and allocates conservatively). On
data-structure work the gap closes: 200k insertions into an
immutable hash — Puffin HAMT 0.08 s vs Racket 0.15 s (user time).
For a teaching-descended compiler with a five-register allocator,
"same order of magnitude as Chez" is a healthy place to bootstrap
from: the compiler compiling itself will be seconds, not minutes.

Compile times: the largest corpus program (puffin-9, the eval/apply
interpreter, with prelude) runs the whole 16-pass pipeline in
**~57 ms** (regalloc 26 ms, select 7 ms, render 9 ms, everything
else ≤5 ms). Compilation cost is a non-issue for bootstrapping.

## 2. Feature inventory: Racket facilities used vs Puffin

Method: an operator census over `src/*.rkt` (grep-based, then
hand-classified). Verdicts: **have** (already in Puffin),
**added** (implemented in this pass — see §3), **idiom** (a
mechanical rewrite pattern, documented below), **defer** (not
needed for a first bootstrap).

| Racket facility | uses | verdict |
|---|---|---|
| `match` w/ quasiquote + `...` (incl. nested) | everywhere | **have** |
| quasiquote *construction* `` `(,x ,@ys) `` | everywhere | **added** |
| internal `define`s in bodies | every pass | **added** |
| `letrec` | few | **added** |
| `gensym` | 52 | **added** |
| `format` | 110 | **added** (args as a list) |
| immutable hash/set + algebra (`set-union` …) | ~90 | **have** / **added** (algebra in prelude) |
| `sort`, `symbol<?`, `string<?` | 11 | **added** |
| `apply` | 19 | **added** (≤5 args — the register-argument budget) |
| list utilities (`take drop index-of append-map filter-map first rest findf remove partition`) | ~40 | **added** |
| 2-list `map` | frequent | **added** (`map2`) |
| `number->string` / `string->number` / `substring` / `string-byte` | render + reader | **added** |
| `bitwise-*`, `arithmetic-shift`, `modulo` | tag math | **added** |
| `displayln`, `value->string` | 24 | **added** |
| reading source (`read`, file input) | driver | **added** (`read-all` prim; s-expr reader written in Puffin is the next seed step) |
| multiple values / `define-values` | 33 | **idiom**: return a list, `match` it |
| `for`/`for/list`/`for/hash` … | ~40 | **idiom**: named `let`, `map`, `foldl` |
| `struct` | 2 (`prim-spec`) | **idiom**: tagged lists or vectors |
| parameters / `parameterize` | 16 | **idiom**: top-level define + `set!` (dynamic wind not needed by the compiler) |
| `with-handlers` / exceptions | 3 | **idiom**: the compiler can halt on error (`error`); trace-driver error capture stays host-side |
| hash iteration (`in-hash`) | 3 | **have** (`hash-keys` + `hash-ref`) |
| chars (`char=?` …) | rt-sym only | **idiom**: `string-byte` + integer compares |
| `pretty-print` | debug output | **defer** (plain `println` of s-exprs suffices; the IR printer is host-side) |

Nothing in the compiler proper is outside reach: the remaining
**idiom** rows are local, mechanical rewrites, not language gaps.

## 3. Features implemented in this pass (each tested on all routes)

1. **Quasiquote expressions** with `unquote`/`unquote-splicing`,
   depth-aware (nested quasiquote correct) — desugar's `qq->expr`,
   mirrored in the web interpreter. *The* compiler-construction
   idiom.
2. **Internal defines** — letrec\*-scoped over the whole body
   (mutually recursive inner helpers, exactly how every pass is
   written), via `body->expr`; plus explicit **`letrec`**.
3. **`gensym`** — runtime-interned fresh symbols (`pf_gensym`
   skips taken names, so gensyms survive read-back).
4. **`value->string`** — render any value as `display` would (the
   runtime's display handlers were re-plumbed onto `FILE*` streams
   so rendering into memory is the same code path as printing).
5. **`format`** — in the prelude, over `value->string`; takes its
   arguments as a list (`(format "~a + ~a" (list 1 2))`) because
   Puffin arities are fixed — see "variadics" in §4.
6. **`read-all`** — the rest of stdin as a string: the seed for a
   Puffin-written s-expression reader.
7. **Strings**: `substring`, `string<?`, `string-byte`,
   `number->string`, `string->number`, `string-join` (prelude).
8. **Integer bit work**: `bitwise-and/ior/xor`,
   `arithmetic-shift`, `modulo` (sign follows divisor) — a
   bootstrap backend's tag arithmetic needs these.
9. **`sort`** (stable merge sort), `symbol<?`, plus the list
   utility batch and `map2` — prelude.
10. **Set algebra** over the immutable sets: `list->set`,
    `set-union`, `set-subtract`, `set-intersect` — prelude.
11. **`apply`** for 0–5 arguments (the arity a closure call can
    carry in registers; documented limit with a clear error).

Corpus additions: `puffin-11` exercises every item above;
`puffin-10` is the **bootstrap seed** — `shrink` and a
deterministic `uniqueify` *written in Puffin*, in the house style
(ellipsis quasiquote patterns in, quasiquote construction out,
internal defines for the walkers), run over an embedded program and
byte-identical across all four routes. Fixing what it flushed out
(entry-symbol collisions with pattern variables named `main`;
literal-table scanning of bare quoted-symbol rhs) is exactly why the
seed lives in the test corpus.

## 4. Remaining gaps to full self-hosting (ranked)

1. **An s-expression reader in Puffin** — CLOSED: see
   `puffincc-src/01-reader.puf` (byte-oriented, handles the full
   surface syntax including dotted pairs, the quote family, and
   `#%rest`/`#:when` tokens), exercised by every program puffincc
   compiles.
2. **Variadic functions** — ~~the honest language gap~~ CLOSED:
   dotted formals landed with an arity-register calling convention
   (every closure call passes its logical argument count; variadic
   prologues rebuild the rest list via the runtime's spill area,
   uniformly across the >6-argument overflow protocol). `format` is
   now fully variadic; `apply` covers 0–5 arguments. See puffin-13
   in the corpus and the "Variadic functions" commit.
3. **Output** — the compiler emits one big string today via
   `format`/`string-append`; `display` to stdout suffices for a
   bootstrap driver (`puffin compile.puf < prog.puf > prog.s`).
   File I/O beyond stdin/stdout can stay host-side.
4. **Performance headroom** (optional): string-append is O(n) per
   append (rendering is quadratic-ish in the worst case) — a
   string-builder primitive or rope would help the render pass at
   scale; measured irrelevant at current program sizes.
5. **Arity checks on closure calls** are absent in compiled code
   (the interpreters check) — worth adding before large Puffin
   codebases, independent of bootstrap.

Suggested bootstrap order: reader → irs predicates + desugar/shrink
(pure tree work, already seeded) → middle passes → select/regalloc →
render; drive with the Racket main.rkt harness throughout (it can
run a Puffin-compiled pass binary per stage and diff against the
Racket pass — the golden machinery already knows how to compare).

## 5. SELF-HOSTING: achieved

`puffincc-src/` contains the full compiler in Puffin (~5,200 lines —
a module DAG rooted at main.puf: reader, module resolution, desugar,
the optimizer (contraction + CESK* flow analysis), middle passes,
regalloc, both backends, CLI driver), kept in lockstep with the
reference via generated table modules
(src/gen-puffincc-tables.rkt) and per-pass cross-implementation
diffing (src/diff-ir.rkt + the driver's (dump-after pass)
directive). Results:

- **Stage 1** (hosted build, `bin/build-puffincc`): the entire
  corpus passes through puffincc on arm64 and targeting x86-64
  (98 programs × 3 inputs per route, module programs included).
- **Stage 2**: puffincc compiles itself — reading and resolving its
  own module DAG from disk; the stage-2 binary passes the corpus.
- **Stage 3 fixpoint**: stage-2-compiled puffincc produces
  **byte-identical** assembly to stage 2. Determinism comes free:
  the runtime's gensym counter starts fresh per process.

The C runtime stays host-side by design, but puffincc now drives the
assemble/link step itself: `build/puffincc prog.puf -o prog` reads
the program (resolving its require/provide modules from disk),
compiles, writes the assembly, and shells out to clang via the
`system` primitive — no wrapper script. `-o out.s` stops at
assembly; no `-o` prints it (the classic pipe mode, `puffincc <
prog.puf > prog.s`, also survives for single-file programs).
puffincc itself is a module DAG rooted at `puffincc-src/main.puf`
(docs/MODULES.md), and carries the full optimizer port
(`optimize.puf` = contraction/inlining + the -O1 pass hooks;
`aam.puf` = the -O2 flow analysis; docs/OPTIMIZER.md).

## 6. How this was validated

Every feature landed with: manifest entry (C entry point + Racket
reference implementation + doc line), C implementation in the
pluggable runtime, web-interpreter implementation, corpus test, and
a full `-m all` + chain + `test-corpus.mjs` run. Final state:
**549 + 183 checks, zero failures**, `docs/STDLIB.md` regenerated
from the manifest.
